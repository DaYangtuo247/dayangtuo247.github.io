工时计算器：基于 Function Calling

最近我做了一个自己的考勤工时计算器。

一开始我以为这个需求会很简单：

  • 读取 Excel
  • 提取打卡时间
  • 配对上下班
  • 计算工时

但真正处理真实数据之后，我发现问题根本不在“计算”，而在：

如何判断哪些打卡，才是真正的工作时间。

例如下面这组打卡：

09:19:14 11:52:08 12:44:37 14:13:31 14:36:22 14:41:38 14:44:24 18:03:15 20:03:15

如果纯按规则硬配对，很容易得到：

  • 12:44:37 - 14:13:31
  • 14:36:22 - 14:41:38
  • 14:44:24 - 18:03:15

但真实场景里：

  • 有人会提前回工位
  • 有人会继续午休
  • 有人会中途出去
  • 晚上可能会回来加班

结合真实语义，更合理的判断应该是：

  • 上午：09:19:14 - 11:52:08
  • 下午：14:13:31 - 14:44:24
  • 晚上：18:03:15 - 20:03:15

这个问题本质上已经不是“时间计算”，而是“时间段语义判断”。

所以如果完全依赖固定规则，最后通常会变成大量：

if ...
elif ...
elif ...

规则会越来越复杂，而且很难适配不同人的作息习惯。

后面我换了一个思路：

  • LLM 负责做“时间段语义判断”
  • Function Calling 负责保证结构稳定
  • Excel 负责最终工时计算

整个流程大概是这样：

Excel 原始考勤
    ↓
Python 解析打卡
    ↓
去重清洗
    ↓
LLM 判断真实工作区间
    ↓
Function Calling 输出结构化结果
    ↓
写回 Excel

Prompt

这类问题里，真实样例通常比抽象规则更重要。

我最后主要做了两件事：

  1. 明确中午午休场景
  2. 给模型大量真实样例

例如：

examples = (
    "只参考这些输入/输出模式。下午开始时间指真正开始干活的时间，不是吃饭回来或午休中的打卡。\n"
    "输入：13:56:03 16:02:08 17:35:38 18:35:27 19:50:52\n"
    "输出：上午 空；下午 13:56:03-16:02:08；晚上 17:35:38-19:50:52\n"
    "说明：上午无有效出勤，13:56:03 是真实下午开工。\n"
    "输入：09:19:14 11:52:08 12:44:37 14:13:31 14:36:22 14:41:38 14:44:24 18:03:15 20:03:15\n"
    "输出：上午 09:19:14-11:52:08；下午 14:13:31-14:44:24；晚上 18:03:15-20:03:15\n"
    "说明：14:13:31 更像真正开始工作，18:03:15-20:03:15 更像回来加班。\n"
)

Function Schema

Function schema 我这里尽量保持简单，不做花活。

tools = [
    {
        "type": "function",
        "function": {
            "name": "set_attendance_ranges",
            "description": "根据每天的原始打卡，返回上午、下午、晚上真正的开始和结束时间。",
            "strict": True,
            "parameters": {
                "type": "object",
                "properties": {
                    "days": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "date": {"type": "string"},
                                "morning_start": {"type": "string"},
                                "morning_end": {"type": "string"},
                                "afternoon_start": {"type": "string"},
                                "afternoon_end": {"type": "string"},
                                "evening_start": {"type": "string"},
                                "evening_end": {"type": "string"},
                            },
                            "required": [
                                "date",
                                "morning_start",
                                "morning_end",
                                "afternoon_start",
                                "afternoon_end",
                                "evening_start",
                                "evening_end",
                            ],
                            "additionalProperties": False,
                        },
                    }
                },
                "required": ["days"],
                "additionalProperties": False,
            },
        },
    }
]

这里有两个实践经验：

  1. strict 模式能明显减少输出漂移
  2. 空值尽量直接约定为空字符串，后处理更简单

调用

from openai import OpenAI

client = OpenAI(
    api_key=os.getenv("DEEPSEEK_API_KEY"),
    base_url="https://api.deepseek.com/",
)

response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=[
        {
            "role": "system",
            "content": (
                "你是考勤助手。只判断每天上午、下午、晚上真正的开始和结束时间。"
                "用户中午会吃饭并午休，12:30 到 14:00 之间的打卡经常只是回工位、吃饭回来、准备午睡、睡醒后活动，不一定表示已经开始工作。"
                "下午开始时间必须表示真正开始干活的时间，不能把午休时间算成下午工时。"
                "如果某一时段不存在，返回空字符串。你只能调用函数，不要输出额外解释。"
            ),
        },
        {
            "role": "user",
            "content": examples + "\n请判断下面整个月记录：\n" + "\n".join(lines),
        },
    ],
    tools=tools,
    temperature=0.2,
    max_tokens=12000,
)

然后读取返回：

message = response.choices[0].message

if message.tool_calls:
    arguments = message.tool_calls[0].function.arguments
else:
    arguments = message.content or "{}"

result = json.loads(arguments)

总结

我觉得这套方法特别适合下面这类业务：

  • 固定规则很难覆盖全部边界
  • 但输出结果又必须结构化
  • 用户可以提供足够多的真实例子

比如：

  • 考勤打卡判断
  • 售后工单分类
  • 发票字段抽取
  • 日程语义整理

它们的共同点都是：规则很难完全覆盖，但结果又必须结构化。
