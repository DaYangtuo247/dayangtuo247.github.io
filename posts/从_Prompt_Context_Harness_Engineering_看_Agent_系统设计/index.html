从 Prompt/Context/Harness Engineering 看 Agent 系统设计

参考资料：

  • 最近爆火的 Harness Engineering 到底是啥？一期讲透！

最近在看一些关于 Agent / LLM 系统设计的内容，有一个特别清晰的脉络，可以把当前 AI 工程的发展分成三个阶段：

Prompt Engineering → Context Engineering → Harness Engineering

如果只记一句话的话：

AI 的问题，已经从“模型说得对不对”，变成“系统能不能把事情做对”。

Prompt Engineering

最早大家接触大模型的时候，核心工作其实很简单：写 prompt。

比如：

  • 给模型设定角色（你是一个金融分析师）
  • 给示例（few-shot）
  • 规定输出格式（JSON / Markdown）

本质上你在做的事情是：

用语言去约束模型的输出分布

也就是——让模型更容易“猜对你想要的答案”。

这个阶段的特点很明显：

  • 调整 prompt 就能明显提升效果
  • 很多问题可以靠“提示词技巧”解决
  • 成本低，上手快

但问题也很快出现：Prompt 只能影响“表达”，不能补充“能力”。

比如：

  • 模型不知道的数据，它还是不知道
  • 多步骤任务（比如分析→决策→执行）很容易崩
  • 一旦任务复杂，prompt 会变得非常脆弱

Context Engineering

当大家发现 prompt 不够用了，就进入了第二阶段：Context Engineering。

核心思路变成：

模型不一定要变强，但必须“看到正确的信息”。

于是出现了一整套典型方案：

  • RAG（检索增强）
  • 多文档拼接
  • 工具调用（数据库 / API / 搜索）

这个阶段你做的事情，本质是：

控制模型“输入什么上下文”

模型不再是“凭空回答”，而是基于你喂进去的数据进行推理。

相比 Prompt Engineering，这一步解决了一个关键问题：知识问题被工程化解决了

但新的问题也出现了：

  • 信息给对了 ≠ 结果就对
  • 多步骤任务依然不稳定
  • Agent 很容易“走偏”

换句话说：模型“知道了”，但不一定“做对了”。

Harness Engineering

这就是现在最关键的一步：Harness Engineering。

可以把 Harness 理解成：

一个包在模型外面的“执行系统”

如果用一个更工程的类比：

传统：
请求 → 模型 → 返回结果

现在：
请求 → Harness → 模型 → 校验 → 重试 → 状态更新 → 输出

也就是说，模型不再是系统的核心，而是系统中的一个组件。

Harness 在解决什么问题？

一句话总结：

模型是“会思考的函数”，但它不可靠。

所以 Harness 的目标是：让一个不稳定的模型，稳定地完成任务

它具体做了什么？

其实就是把传统软件工程那一套，搬到了 AI 上：

  • 任务拆解：不让模型一步到位，而是分阶段执行
  • 状态管理：记录当前进度，而不是每次重新推理
  • 过程校验：每一步结果都要检查
  • 错误恢复：失败可以 retry / fallback
  • 流程控制：分支、循环、条件执行

你可以把它理解为：

给大模型套了一层“调度系统 + 状态机 + 容错机制”

一个很关键的认知变化

在这个阶段，性能提升的来源已经变了：

以前：

  • 换更强的模型
  • 写更好的 prompt

现在：

  • 改执行流程
  • 加校验机制
  • 做重试策略

甚至在一些实际案例里：模型完全不变，只改 Harness，成功率可以从 60% 提到 90%+

对比

阶段核心问题解决思路技术重点局限性
Prompt Engineering模型是否听懂指令优化语言表达角色设定、风格约束、示例引导无法补充知识缺失，不擅长管理动态信息
Context Engineering模型是否获得正确信息优化信息供给检索增强（RAG）、流式数据流、信息分层无法解决执行过程中的监督与纠偏问题
Harness Engineering模型能否稳定执行任务优化运行系统执行编排、状态管理、错误恢复-

成熟 Harness 的六层架构

一、上下文边界层

  • 核心功能：确保模型在正确边界内思考

关键组件：

  1. 角色与目标定义：明确模型身份、任务范围及成功标准
  2. 信息裁剪与变换：确保上下文相关性（非越多越好）
  3. 结构化组织：分层管理规则、任务状态、外部证据

二、工具系统层

  • 核心功能：连接模型与现实世界的桥梁

关键模块：

  1. 工具选择：平衡能力覆盖与使用复杂度
  2. 调用机制：避免不必要调用（不乱查）和错误判断（不乱答）
  3. 结果处理：提炼工具返回结果，保持与任务相关性

三、执行编排层

  • 核心功能：将任务分解为可执行步骤

典型流程： 目标理解 → 信息检索 → 分析处理 → 输出生成 → 结果检查 → 修正迭代

价值：解决 Agent “想到哪做到哪”的无序执行问题，确保任务闭环

四、记忆与状态层

  • 核心功能：解决 Agent “失忆”问题

状态分类：

  1. 当前任务状态
  2. 会话中间结果
  3. 长期记忆与用户偏好

管理原则：分类存储，避免信息混乱导致执行偏差

五、评估与观测层

  • 核心功能：建立质量反馈机制

关键组件：

  1. 输出校验与验收
  2. 自动化测试系统
  3. 日志与指标监控
  4. 错误归因分析

价值：避免 Agent “自我感觉良好”的认知偏差

六、约束校验与恢复层

  • 核心功能：保障系统鲁棒性

三大机制：

  1. 约束机制：明确能力边界（能做什么 / 不能做什么）
  2. 校验机制：输出前后的检查流程
  3. 恢复机制：失败后的重试、回滚策略

重要性：真实环境中失败是常态，恢复能力决定系统可用性

一线公司实践案例

1. Anthropic 的自编排系统

核心问题1：长任务上下文过载

  • 解决方案：Context Reset（上下文重置），通过新的上下文重新引导模型，避免信息过载导致的性能下降，类似新老员工交付
  • 类比：内存泄漏后重启进程并清理缓存

核心问题2：自评失真

  • 解决方案：角色分离架构
    • Planner（规划者）：需求拆解与规格
    • Generator（生成者）：逐步实现
    • Evaluator（评估者）：环境化测试（真实操作验证）

2. OpenAI 的 Agent 开发实践

工程师角色转变：从写代码 → 设计环境

关键策略：

  1. 渐进式检索：将巨型文档拆分为目录 + 子文档，按需加载
  2. 环境化验证：Agent 通过测试（截图 / 操作 / 日志系统）验证环境
  3. 自动治理系统：将专家工程经验编码为可执行规则（含修复方案）
