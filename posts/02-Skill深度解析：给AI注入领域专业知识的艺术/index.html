Skill 深度解析：给 AI 注入领域专业知识的艺术

系列文章：本文是「MCP 与 Skill 技术体系」三篇系列的第二篇，聚焦于 Claude Code Skill 的设计思想与工程实践。

一、Skill 解决什么问题？

1.1 LLM 的“知识边界”困境

大语言模型拥有海量通用知识，但在具体项目里，经常会出现一种尴尬情况：它很聪明，却不够“懂你”。

例如：

  • 不了解你的代码库约定：团队要求所有 API 错误必须返回 { code, message, data }，但模型并不知道；
  • 不懂你的领域术语：公司内部把“归因分析”叫作 attribution，把某个内部框架叫 Phoenix，模型一开始没有这些背景；
  • 无法复用专家经验：资深工程师总结了一套代码审查 checklist，但每次对话都要重新粘贴；
  • 风格不稳定：注释风格、变量命名、测试结构、提交说明，每次生成都可能不一样。

这些问题不是 MCP 主要解决的。

MCP 更偏向解决：AI 能访问哪些外部工具和数据源。

比如：

  • 读取数据库；
  • 调用 GitHub；
  • 查询文件系统；
  • 操作浏览器；
  • 调用公司内部 API。

而 Skill 更偏向解决：AI 在特定场景下应该如何思考、如何行动、遵守什么规范。

一句话概括：

MCP 给 AI 接工具，Skill 给 AI 注入专业工作方法。

1.2 Skill 的本质

Claude Code 的 Skill，本质上是一种可复用的专业能力包。

更准确地说，一个包含 SKILL.md 的目录就是一个 Skill。它可以很轻，也可以很重：轻量 Skill 可能只有几十行规则；复杂 Skill 则可以附带参考文档、示例、脚本和静态资源。

Skill 的几个典型特点是：

  • 可复用：一次编写，以后在相同任务里反复使用，不需要每次重新粘贴提示词；
  • 可分享：本质是文件目录，可以通过 Git、插件市场或团队仓库共享；
  • 可自动触发：Claude Code 会根据 description 判断当前任务是否匹配某个 Skill；
  • 也可手动触发：当你明确知道要使用某个 Skill 时，可以用斜杠命令调用。

二、Claude Code 的 Skill 体系

Claude Code 的 Skill 是以 SKILL.md 为核心的目录结构。根据作用范围不同，常见位置可以分为几类：

# 用户级 Skill：对所有项目生效，适合个人常用习惯
~/.claude/
└── skills/
    └── skill-name/
        └── SKILL.md

# 项目级 Skill：只对当前项目生效，适合团队共享
.claude/
└── skills/
    └── skill-name/
        └── SKILL.md

# 插件级 Skill：随插件安装而生效
my-plugin/
└── skills/
    └── skill-name/
        └── SKILL.md

如果是企业环境，还可能存在企业级托管配置。整体可以理解为四层：

层级常见位置适用场景
企业级Managed Settings公司统一规范，全员生效
用户级~/.claude/skills/个人常用 Skill，比如 commit message、代码审查
项目级.claude/skills/项目专属规范，比如 API 风格、部署流程、数据库约定
插件级<plugin>/skills/随插件安装的一组 Skill

需要注意的是，同名 Skill 出现在多个层级时会有优先级问题。实践中可以按这个原则理解：越靠近组织治理和个人配置的 Skill 越容易覆盖通用 Skill；而项目级 Skill 更适合沉淀仓库内的团队规范。

对于 Monorepo，也可以在子项目中放置嵌套的 .claude/skills/，让不同模块使用不同规范：

monorepo/
├── frontend/
│   └── .claude/
│       └── skills/
│           └── react-style/
│               └── SKILL.md
├── backend/
│   └── .claude/
│       └── skills/
│           └── api-style/
│               └── SKILL.md
└── .claude/
    └── skills/
        └── project-overview/
            └── SKILL.md

一个完整的 Skill 目录通常长这样：

.claude/
└── skills/
    └── sql-review/
        ├── SKILL.md              # 必需：Skill 入口文件
        ├── references/           # 可选：详细参考资料
        │   ├── schema.md
        │   └── query-style.md
        ├── scripts/              # 可选：可执行脚本
        │   └── check_sql.py
        ├── examples/             # 可选：正反示例
        │   ├── good.sql
        │   └── bad.sql
        └── assets/               # 可选：图片、模板、静态资源
            └── er-diagram.png

其中只有 SKILL.md 是必需的，其余目录按需添加。

三、SKILL.md 文件结构

3.1 最小可用结构

一个最简单的 Claude Code Skill 只需要一个 SKILL.md：

---
name: code-review
description: Use this skill when reviewing code changes, checking bugs, maintainability, security issues, and test coverage.
---

# Code Review Skill

## Review Principles

When reviewing code, focus on the following areas:

1. Correctness
2. Security
3. Maintainability
4. Performance
5. Test coverage

## Output Format

Use this structure:

- Summary
- Major Issues
- Minor Suggestions
- Suggested Fixes

这个文件分为两部分：

  1. YAML Frontmatter：name、description 等元数据；
  2. Markdown Body：真正的工作规范和执行流程。

其中最重要的是 description。

Claude Code 会根据 description 判断：当前任务是否应该使用这个 Skill。

3.2 name 与 description

name 是 Skill 的名称，通常使用短横线命名：

name: code-review

description 决定 Skill 何时被自动触发，应该写得具体、明确、可匹配任务场景。它是自动匹配的关键字段，建议把最重要的触发关键词放在前面，因为实际用于匹配的描述内容存在长度限制：

description: Use this skill when the user asks to review code, inspect a pull request, find bugs, check maintainability, or evaluate security and test coverage.

如果场景比较复杂，可以增加 when_to_use，把触发条件写得更具体：

name: commit-msg
description: Generate a conventional Git commit message for staged or unstaged code changes.
when_to_use: |
    - 用户要求生成 commit message、提交信息、git commit 文案
    - 用户准备提交代码，需要总结当前 diff
    - 用户要求遵循 Conventional Commits 规范

description 不要写成“帮助写代码”这种泛泛描述，否则很难精准触发。更好的写法是直接说明任务、关键词和适用边界。

3.3 Skill 内容主体

SKILL.md 的正文应该像一份工程手册，而不是泛泛而谈的提示词。

例如，一个后端 API Skill 可以这样写：

---
name: backend-api-style
description: Use this skill when creating, modifying, or reviewing backend API endpoints in this project. Covers FastAPI, Pydantic models, error response format, and repository-layer conventions.
---

# Backend API Style

## Tech Stack

- Python 3.11+
- FastAPI
- Pydantic v2
- SQLAlchemy 2.0 async mode
- PostgreSQL

## API Response Format

All successful responses must use this structure:

```json
{
	"code": 0,
	"message": "success",
	"data": {}
}
```

For errors:

```json
{
	"code": 1001,
	"message": "user not found",
	"data": null
}
```

## Repository Rules

- Do not write SQL directly inside route handlers.
- Use Repository classes for database access.
- Use `async with session` for database operations.
- Avoid committing inside loops.

## Validation Rules

- All request bodies must use Pydantic models.
- Do not trust raw user input.
- Do not concatenate SQL strings manually.

## Before Finishing

Check:

- Whether the response format is correct.
- Whether input validation is complete.
- Whether database access is isolated in the Repository layer.
- Whether errors use the project error code convention.

这类 Skill 的价值在于：以后你让 Claude Code 写接口、改接口、审查接口，它就能自动带上这套项目规范。

3.4 Frontmatter 常用字段

claude code 官方参考文档：https://code.claude.com/docs/zh-CN/skills

除了 name 和 description，Skill 还可以通过 Frontmatter 控制触发方式、工具权限和运行策略。下面这些字段比较常用：

字段作用典型用法
nameSkill 名称，用于识别和手动触发name: code-review
description自动触发匹配依据描述任务场景和关键词
when_to_use补充触发条件写具体用户话术和适用边界
allowed-tools指定 Skill 可直接使用的工具部署检查、文件读取、脚本执行
disable-model-invocation禁止自动触发，只允许手动调用生产部署、数据库迁移、发送邮件
user-invocable是否允许用户从 / 菜单手动调用背景知识型 Skill 可设为 false
paths只在匹配特定文件路径时触发API 文档、路由文件、迁移脚本
effort指定思考力度复杂审查可设为 high
context / agent在独立子代理中运行大型任务、隔离上下文

例如，一个只希望手动触发的生产部署 Skill 可以这样写：

---
name: deploy-to-production
description: Deploy the application to production. Use only when the user explicitly requests production deployment.
disable-model-invocation: true
allowed-tools: ["Bash", "Read"]
effort: high
---

这样做的意义是：部署、删库、发邮件、提交到主分支这类有副作用的操作，不应该被模型因为语义匹配而自动触发，必须由用户明确调用。

四、渐进式加载

Skill 不是一股脑全部塞进上下文。

Claude Code 的设计更接近“渐进式加载”：

启动 Claude Code
│
▼
扫描可用 Skill
│
├── 读取 Skill 名称
├── 读取 Skill description
└── 不立即读取完整正文和资源文件
│
▼
用户提出任务
│
▼
Claude Code 判断任务是否匹配某个 Skill
│
▼
匹配成功后加载对应 SKILL.md
│
▼
如有必要，再读取 references/、examples/、scripts/ 等辅助文件
│
▼
按照 Skill 规范执行任务

这个机制的核心好处是：

  • 平时只加载 Skill 的简短描述，节省上下文；
  • 真正需要时才加载完整规范；
  • 大型参考资料可以放到 references/，需要时再读取；
  • 固定流程可以放到 scripts/，避免模型每次重复生成脚本。

所以写 Skill 时，不要把几万字规范都塞进 SKILL.md。

SKILL.md 应该像目录和任务手册，而不是完整知识库。

更好的方式是：

SKILL.md               # 放核心原则、触发场景、执行流程
references/*.md        # 放详细规范、背景知识、长文档
examples/*             # 放正反示例
scripts/*              # 放可执行、可复用、确定性的脚本

五、安装、触发与管理

5.1 安装别人的 Skill

安装 Skill 最简单的方式是复制整个目录，而不是只复制 SKILL.md：

# 安装到用户级，所有项目可用
mkdir -p ~/.claude/skills
cp -r skill-name ~/.claude/skills/

# 安装到项目级，只在当前项目可用
mkdir -p .claude/skills
cp -r skill-name .claude/skills/

目录结构必须是：

~/.claude/skills/my-skill/SKILL.md

而不是：

~/.claude/skills/SKILL.md

如果 Skill 来自插件市场，也可以通过 Claude Code 的插件命令安装。例如先添加市场，再安装插件：

/plugin marketplace add anthropics/skills
/plugin install document-skills@anthropic-agent-skills

这类插件通常不是只包含一个 Skill，而是一组 Skill、命令、hooks 或 MCP 配置的集合。

5.2 创建自己的 Skill

自己写 Skill 不需要复杂编程能力。最低成本的做法是创建目录和 SKILL.md：

mkdir -p .claude/skills/commit-msg
vim .claude/skills/commit-msg/SKILL.md

然后写入：

---
name: commit-msg
description: Generate a conventional Git commit message when the user asks for commit message, git commit text, or code change summary.
---

# Commit Message Skill

根据当前代码变更生成提交信息。

## Format

- 使用 Conventional Commits 风格
- 第一行不超过 50 个字符
- 如果变更较多，在正文中补充 2-3 条说明
- 优先使用中文描述

## Types

- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建、依赖、工具链

也可以直接让 Claude Code 帮你生成：

帮我创建一个用于生成 REST API 文档的 Skill，要求包含接口说明、参数表、返回值示例、错误码说明，并保存到 .claude/skills/api-doc/SKILL.md

相比“帮我写一个文档 Skill”，这种描述更容易生成高质量结果，因为它明确了用途、输出结构和保存位置。

5.3 自动触发与手动触发

Skill 有两种主要调用方式。

自动触发依赖 description 和 when_to_use。当你说“帮我 review 这段代码”时，Claude Code 会从已安装 Skill 的描述中判断是否应该加载 code-review。

手动触发则使用斜杠命令：

/commit-msg
/code-review
/fix-issue 123

如果不知道当前有哪些 Skill，可以使用：

/skills

或者输入 / 打开命令面板进行筛选。

5.4 控制触发行为

有些 Skill 适合自动触发，比如代码风格、文档规范、测试规范；有些 Skill 则不适合自动触发，比如生产部署、数据库迁移、批量删除文件。

可以用下面两个字段控制：

disable-model-invocation: true
user-invocable: false

常见组合如下：

配置用户可手动触发Claude 可自动触发适用场景
默认是是普通 Skill
disable-model-invocation: true是否有副作用操作，只允许手动调用
user-invocable: false否是背景知识型 Skill，不需要用户手动点选
两者都设置否否临时禁用或废弃 Skill

5.5 管理和排查

常用管理命令：

/skills     # 查看已安装 Skill
/plugin     # 管理插件
/cost       # 查看当前会话 token 消耗
/clear      # 清理当前上下文

如果 Skill 没有触发，优先检查：

  • SKILL.md 是否放在正确目录；
  • description 是否太泛；
  • 是否需要补充 when_to_use；
  • 是否误设置了 disable-model-invocation: true；
  • 是否安装了多个同名 Skill，导致实际生效的不是你以为的版本。

六、核心文件与目录详解

6.1 SKILL.md：技能入口

SKILL.md 是整个 Skill 的入口，也是 Claude Code 首先读取的文件。

它应该回答三个问题：

  1. 这个 Skill 什么时候使用？
  2. 使用时应该遵循什么流程？
  3. 需要进一步信息时，该读哪些文件或运行哪些脚本？

例如：

---
name: data-analysis
description: Use this skill when analyzing datasets, performing EDA, cleaning data, building baseline models, or summarizing findings from notebooks and CSV files.
---

# Data Analysis Skill

## Workflow

1. Understand the dataset and task goal.
2. Inspect shape, columns, missing values, and data types.
3. Identify target variable and feature columns.
4. Perform exploratory analysis.
5. Build a simple baseline before complex models.
6. Summarize findings clearly.

## Reference Files

- For EDA checklist, read `references/eda-checklist.md`.
- For visualization style, read `references/visualization-style.md`.
- For reusable scripts, check `scripts/`.

6.2 references/：详细知识库

references/ 适合放长文档。

比如：

references/
├── api-error-codes.md
├── database-schema.md
├── architecture.md
└── test-guidelines.md

SKILL.md 中只需要写清楚什么时候读取它们：

## Additional References

- When modifying API error handling, read `references/api-error-codes.md`.
- When changing database queries, read `references/database-schema.md`.
- When writing tests, read `references/test-guidelines.md`.

这样可以避免 SKILL.md 过长。

6.3 examples/：正反示例

模型通常非常依赖示例。

如果只写“代码要清晰”，效果很一般；如果给出好代码和坏代码，效果会明显更好。

examples/
├── good_repository.py
├── bad_repository.py
├── good_test.py
└── bad_test.py

在 SKILL.md 中可以这样引用：

## Examples

Before implementing repository-layer changes, compare against:

- `examples/good_repository.py`
- `examples/bad_repository.py`

6.4 scripts/：确定性工具

有些事情不适合靠模型“猜”，更适合写成脚本。

例如：

  • 检查 SQL 是否有危险语句；
  • 统计测试覆盖率；
  • 校验 Markdown frontmatter；
  • 生成固定格式的报告；
  • 检查某些命名规范。

目录示例：

scripts/
├── validate_frontmatter.py
├── check_sql.py
└── generate_report.py

SKILL.md 可以告诉 Claude Code：在需要确定性校验时运行脚本，而不是自己凭感觉判断。

七、实战案例：代码审查 Skill

假设我们要为项目创建一个代码审查 Skill。

目录结构：

.claude/
└── skills/
    └── code-review/
        ├── SKILL.md
        ├── references/
        │   └── review-checklist.md
        └── examples/
            ├── good-review.md
            └── bad-review.md

SKILL.md 可以这样写：

---
name: code-review
description: Use this skill when reviewing code changes, pull requests, diffs, or modified files. Focus on correctness, maintainability, security, performance, and test coverage.
---

# Code Review Skill

## Review Goal

Review the code as a senior engineer. Prioritize issues that affect correctness, security, maintainability, performance, and test reliability.

Do not only summarize the code. Find concrete problems and propose actionable fixes.

## Review Process

1. Understand the intent of the change.
2. Inspect the changed files and related context.
3. Identify correctness issues first.
4. Check security and input validation.
5. Check maintainability and code structure.
6. Check test coverage.
7. Provide concise, actionable feedback.

## Severity Levels

Use these severity levels:

- `blocker`: must fix before merge
- `major`: should fix before merge
- `minor`: improvement suggestion
- `nit`: style or readability issue

## Output Format

```markdown
## Summary

Briefly describe what this change does.

## Findings

### 1. [major] Problem title

- Location: `path/to/file.py`
- Problem: explain the issue
- Impact: explain why it matters
- Suggested fix: provide a concrete fix

## Tests

Mention whether existing tests are enough and what additional tests are needed.
```

## Additional References

If the review involves project-specific conventions, read:

- `references/review-checklist.md`

If examples are needed, compare against:

- `examples/good-review.md`
- `examples/bad-review.md`

这个 Skill 比单纯说“帮我 review 一下代码”稳定得多。因为它把评审维度、严重级别、输出格式都固定下来了。

八、Skill 与 MCP 的关系

Skill 和 MCP 经常一起出现，但它们不是同一个层面的东西。

对比项SkillMCP
核心作用注入专业知识、流程、规范接入外部工具和数据源
主要形态SKILL.md + 资源文件MCP Server + Tools/Resources
解决问题AI 应该怎么做AI 能调用什么
典型例子代码审查规范、SQL 优化流程、论文阅读流程访问数据库、读 GitHub、查文件、调用浏览器
是否偏静态偏静态偏动态

可以这样理解：

Skill 告诉 Claude Code：遇到这个任务时，应该按照什么专业流程做。
MCP 告诉 Claude Code：如果需要外部能力，可以调用哪些工具。
Plugin 则更像一个工具箱，可以把多个 Skill、hooks、agents、MCP servers 打包在一起分发。

所以三者不是互斥关系：

  • Skill 是单个能力模块；
  • Plugin 是能力集合和分发容器；
  • MCP 是外部工具和数据连接协议。

在实际项目里，一个 Plugin 可以包含多个 Skill；一个 Skill 可以规定流程，并在执行过程中调用 MCP 提供的工具。

九、推荐 Skill 体系

对于一个中大型项目，可以这样组织：

.claude/
└── skills/
    ├── project-overview/
    │   ├── SKILL.md
    │   └── references/
    │       └── architecture.md
    │
    ├── code-review/
    │   ├── SKILL.md
    │   └── references/
    │       └── review-checklist.md
    │
    ├── backend-api-style/
    │   ├── SKILL.md
    │   ├── references/
    │   │   ├── error-codes.md
    │   │   └── api-style.md
    │   └── examples/
    │       ├── good_api.py
    │       └── bad_api.py
    │
    ├── database/
    │   ├── SKILL.md
    │   ├── references/
    │   │   ├── schema.md
    │   │   └── migration-rules.md
    │   └── scripts/
    │       └── check_migration.py
    │
    └── testing/
        ├── SKILL.md
        └── references/
            └── test-style.md

每个 Skill 只负责一个专业领域，Claude Code 在需要时再加载对应能力。

十、总结

Claude Code Skill 的核心价值，不是让模型“突然变聪明”，而是让模型在你的项目里变得更专业。

它把原本散落在团队经验、代码审查意见、项目文档里的知识，整理成 Claude Code 可以自动使用的能力包。

没有 Skill 时：

  • 每次都要重复说明项目规范；
  • AI 输出风格不稳定；
  • 老经验很难复用；
  • 项目越复杂，沟通成本越高。

有了 Skill 后：

  • 规范可以自动注入；
  • 专家经验可以沉淀；
  • 团队可以共享同一套 AI 工作方式；
  • Claude Code 可以按项目标准完成任务。
  • 还可以通过手动触发、路径匹配和触发开关控制风险。

如果只想记住几个工程实践原则，可以记这几条：

  • 通用习惯放用户级 Skill，项目规范放项目级 Skill；
  • description 决定自动触发质量，不要写得太泛；
  • 有副作用的 Skill 使用 disable-model-invocation: true；
  • 长文档放 references/，正反例放 examples/，确定性逻辑放 scripts/；
  • 定期用 /skills 和 /cost 检查安装情况与 token 消耗。

MCP 让 AI 连接世界，Skill 让 AI 学会你的工作方法。
