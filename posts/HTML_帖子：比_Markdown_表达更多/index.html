HTML 帖子：比 Markdown 表达更多

主题指南 # 主题功能# HTML
Jul 6, 2026
21 min read
—

Markdown 写叙述很顺手，但一旦想表达统计卡片、分类色块图、多列对比这类结构，就只能靠贴图或者放弃。这篇文章本身就是解决方案的演示：正文仍然是 markdown，视觉结构由内嵌的 HTML 块表达——样式全部继承站点主题变量，所以暗色模式、移动端、目录、锚点都和普通文章行为完全一致。下面挑了几种最常用的组件秀一下效果，不是完整文档。

写法很简单：用 /html-to-post 触发 Claude Code 里的同名 skill，附上你的内容素材，产出的 .md 文件校对后放进 src/content/ 即可发布。

统计卡片

叙述里出现一组关键数字时，比起写成一句话，卡片一目了然：

0 新增管线代码
1 规则文件
2 套主题自动适配
100% TOC / 锚点兼容

统计卡片：display: grid + 主题表面色，移动端自动两列。

流程图（内联 SVG）

流程图不再需要 mermaid 或截图，内联 SVG 用 var(--theme-*) 着色，暗色模式下自动换色：

三步发布流程：描边、文字、强调色全部来自主题变量。

环形图（SVG，多色分段）

同一套 SVG 描边技巧还能画环形占比图，每个分段一个色相，和下面”分类色块图”用的是同一组颜色，呼应统一的视觉语言：

4 类组件
Markdown 原生 40% 图表 / 流程图 25% 信息卡片 20% 交互组件 15%

四段环形图：每段 stroke-dasharray 按百分比算长度，stroke-dashoffset 累加前面段的长度首尾相接。

迷你趋势图（SVG 折线 + 渐变填充）

数据趋势用一条折线 + 渐变填充表达，比一句”持续增长”直观得多：

这篇文章的组件数量（近 6 次修订） 13

折线用主题强调色，填充渐变从 32% 透明度淡出到 0——stop-color 直接写 var(--theme-accent) 就能让 SVG 渐变跟随主题切换。

对比图表（纯 CSS）

不需要图表库，几行 CSS 就能表达同一个指标的递进对比：

Markdown 叙述
+ Mermaid + 节点图
+ HTML 块 任意视觉结构

表达能力对比：条形图的填充色用 color-mix 从强调色派生深浅，同一个色相表示同一个指标。

分类色块图（保留原始多色语言）

如果原始素材给不同类别分配了不同色相（比如”输入 / 处理 / 输出”三个阶段各自一个颜色），规则要求保留这些色相本身，不能为了”用主题变量”就把它们全部塌缩成一个强调色的深浅变化——那样会丢掉设计想传达的类别区分。做法是把每个分类色相通过 color-mix 混进主题表面色，背景深浅跟着亮暗模式走，但色相不变：

转换
视觉结构包成 .hp 前缀的 HTML + 主题化样式
校验
astro check + 暗色模式 + 移动端截图核对

五个阶段各自一个色相，切到暗色模式看一眼——背景变深了，但蓝、紫、青、珊瑚、绿五种颜色本身没有变成同一种紫色的深浅。

热力网格（同一色相的强度阶梯）

和上面”分类色块图”用多个色相区分类别不同，这种场景只有一个维度（强度高低），用同一色相的深浅阶梯表达最合适——类似 GitHub 的贡献热力图：

三周内容更新热度（示意）
少 多

21 个格子、5 级强度，全部是 color-mix(in srgb, 色相 N%, var(--theme-surface)) 换个百分比——不需要脚本，纯 CSS 就能表达强度渐变。

截图式卡片（固定配色，不跟随站点主题）

有一类内容例外：如果 HTML 块是在还原一张真实截图（终端输出、某个 App 的界面），它就该保持截图本来的颜色，不接主题变量——毕竟真实的终端软件配色不会因为你现在开的是亮色浏览器就变白：

~/devolio-blog-theme
$ pnpm astro check
✓ Result (105 files): 0 errors, 0 warnings
$ _

这个终端窗口在亮色模式和暗色模式下长得一模一样——因为它本身就是在模拟一张截图，不是主题化内容。

警示卡片（比”建议”更强调的徽标样式）

普通的建议用 markdown 引用块就够（见下一节），但如果想要更强的视觉分量——比如标出一类常见误区——可以用带徽标的卡片：

✕ 误区
把整段表格写成自定义 HTML

站点默认的 markdown 表格样式已经够用，自定义 HTML 表格反而要多写一大堆 CSS 才能达到同样效果，得不偿失。

→ 换成：markdown 表格 | a | b |

徽标 + 箭头结构，比纯文字警示更容易一眼扫到。

算法可视化（交互脚本，数据驱动配色）

规则也允许克制的交互脚本——比如冒泡排序的动态演示。每根柱子按自己的数值算一个色相（数值越小越偏蓝青、越大越偏红橙），排序完成后自然呈现一条彩虹渐变，比单色柱子更好看也更直观；脚本被约束在自己的容器里，站内导航来回切换后依然可用：

就绪

冒泡排序可视化：柱子颜色由数值映射到色相（hsl(220 − v·1.9, 65%, 55%)），高亮当前比较的两根柱子，已排定的部分保持全亮度、未排定的部分微微调暗。

什么时候还是纯 Markdown

上面这些花样都是给”markdown 表达不出来的结构”用的。但大部分内容根本用不上 HTML——表格、建议、代码这三类永远是 markdown，直接原生语法写就是最好的效果，理由也简单：站点默认样式已经够好看，自定义反而更丑更难维护。

内容类型用什么
数据行列对比markdown 表格
建议 / 提示markdown 引用块
代码示例markdown 代码块（自带语法高亮）
图片 + 图注markdown ![]() + 斜体说明段落

建议：拿到一份 HTML 素材时先过一遍判断表——能用 markdown 原生语法表达的（标题、表格、图片、引用、代码），就用 markdown；只有图表、多色分类卡片、进度条、截图式 mockup、交互这类 markdown 语法表达不出来的视觉结构，才值得包一层自定义 HTML。

这篇文章怎么来的

上面所有 HTML 视觉块遵循同一份约定：最外层 <div class="hp">、样式集中在文末的 <style> 块且选择器全部以 .hp 开头、颜色只用主题变量（分类色相例外，用 color-mix 保留色相；截图 mockup 例外，固定原色）、章节标题全是 markdown 的 ##（所以右侧目录正常工作）。交互脚本带 data-astro-rerun 并做了幂等守卫，只查询自己容器内的 DOM。

完整规则见仓库内 .claude/skills/html-to-post/SKILL.md——那份文件就是喂给 Claude 的全部内容。
