驯服 Claude Code：七种引导它行为的方式

人工智能工程实践 # 开发工具# ClaudeCode# Skills
Jul 2, 2026
39 min read
—

CLAUDE.md、rules、skills、subagents、hooks、output styles，还有 append-system-prompt —— 这七种机制看起来都是”给 Claude 下指令”，但它们何时加载、会不会在长会话中被压缩丢弃、指令的”权威性”有多强，其实完全不同。分不清楚，轻则浪费上下文，重则关键规则在长会话里悄悄失效。

先搞懂：一次请求里到底有什么

在讲七种引导方式之前，先弄清楚一个更底层的问题：你每按一次回车，Claude 实际收到的是什么。这决定了后面所有机制”扛不扛得住压缩”。

一次完整请求 · 从上到下，是模型实际收到内容的顺序
系统提示词（固定不变）
身份 / 规则 / 内置工具说明（Bash、Read、Edit…）末尾会追加 --append-system-prompt 的内容
常驻目录（只有名称/描述，不含正文）
Skills：仅名称+描述常驻，调用时才加载正文；Subagents：仅名称+描述+工具权限常驻；MCP 工具：仅工具名称常驻，用到时才展开定义
CLAUDE.md 内容
作为第一条"用户消息"，随请求一起发送
之前所有轮次的问答
含被调用过的 skill 正文、subagent 的最终摘要。会话越长，这块越大——直到触发压缩
你刚才这句输入
追加在整个请求的最后
↑ 你按下回车，新的一句话追加进最后一块
↻ 下一轮：这次的回复也会变成"历史对话"的一部分，然后连同前面全部内容，再整体重新发送一次
系统固定 常驻目录（仅索引） CLAUDE.md 对话历史 本次输入

真实案例更直观。下面是 Claude Code 里 /context 命令的实际输出——正好能和上面的图一一对应上：

Context window 58.7k / 200.0k (29%)
Messages36.3k18.2%
System tools8.1k4.1%
System prompt7.3k3.6%
Memory files5.0k2.5%
Skills2.0k1.0%
Autocompact buffer33.0k16.5%
Free space108.3k54.1%
System tools (deferred)15.6k—
Messages（珊瑚色）对应上图里的"对话历史"；System prompt + System tools（蓝色）对应"系统提示词"；Skills（紫色）是"常驻目录"里被单独计费的一项——注意它没有并入 Messages，而是自成一行，调用过的 skill 正文也记在这里，不记进对话历史。System tools (deferred) 是还没展开的 MCP 工具定义，用到时才会真正占用空间，所以没算进上面的百分比里。

为什么要分清这七种方式

每种方式都要回答三个问题：什么时候加载进上下文？长会话被压缩（compaction）时会不会保留？它对 Claude 的约束力有多强？ 下面这张表是速查版，后面按顺序逐个展开说明。

速查表

纵览全局，细节在下面每个小节里详细讲。

方式加载时机压缩后表现上下文成本适合场景
CLAUDE.md（根目录）会话开始，全程常驻被记忆化，压缩后重新读取高 —— 不管用不用都占 token构建命令、目录结构、代码规范、团队约定
CLAUDE.md（子目录）读取该子目录下文件时按需加载离开该目录后即丢失，需重新触发低 —— 只在相关时占用特定子目录专属约定
Rules全局规则在会话开始加载；路径限定规则在触碰匹配文件时加载压缩后会重新注入中 —— 未限定路径则常驻具体约束/规范（如”所有 API handler 必须用 Zod 校验输入”）
Skills会话开始只加载名称+描述；调用时才加载完整内容被调用过的 skill 在压缩后按共享预算重新注入，最旧的先被丢弃低 —— 完整内容仅在调用时加载流程化工作（部署清单、发布检查表）
Subagents会话开始只加载名称、描述、工具列表；通过 Agent 工具调用时才加载正文只有最终摘要消息+元数据会返回主会话低 —— 调用前零成本，运行在独立上下文窗口需要隔离运行、只需要结果摘要的并行/旁支任务
Hooks在生命周期事件触发时执行完全绕过压缩机制低 —— 配置本身在主上下文之外确定性自动化：跑 linter、完成后发 Slack 通知、拦截危险命令
Output Styles会话开始，注入系统提示词永不压缩高 —— 但会覆盖默认系统提示词大幅度角色切换（从”代码助手”变成”通用助手”）
Append System Prompt会话开始，作为 CLI 参数传入永不压缩，仅对当次调用生效中等 —— 会话内首次请求后被缓存语气、回复长度、格式偏好

逐个拆解

七种方式的具体行为、使用建议，以及可以直接抄的示例。

01 · CLAUDE.md 文件

项目根目录下的一个 markdown 文件，会话开始就加载，并在整个会话期间常驻上下文。构建命令、目录结构、monorepo 布局、代码规范、团队约定都适合放在这里。

它其实分两种：

  • 始终加载：根目录的 CLAUDE.md（可以是共享仓库里的，也可以是你本地个人偏好的版本）。会话开始就加载，长会话中不会丢失或退化——压缩发生时 Claude Code 会重新读取它。
  • 按需加载：位于会话初始化目录以下的子目录中的 CLAUDE.md。比如 app/api/CLAUDE.md 只有在 Claude 读取 app/api 目录下的文件时才会加载，而不是会话一开始。它和”路径限定规则”的压缩表现一样：一旦离开那个子目录就会丢失，得重新触碰才会再加载。

当前工作目录下的所有子目录 CLAUDE.md，在 Claude 读取该目录内文件时才会加载。

在共享仓库里，CLAUDE.md 很容易变成”没人管的配置文件”：每个团队都往里加东西，却没人删——规模越大，代价越高。每一行都会加载进每个工程师的每次会话，不管跟他当前任务相不相关，这既消耗 token，也稀释了 Claude 对真正重要指令的遵循程度。文件变大后，应该把”团队专属约定”移到路径限定的 rules 里，把”操作流程”移到 skills 里，让它们只在相关时才加载。

加载时机
会话开始 / 触碰子目录时
压缩表现
根目录会重读；子目录会丢失
上下文成本
高（根）/ 低（子目录）

建议：把根 CLAUDE.md 控制在 200 行以内，指定负责人，像对待代码一样 review 它的变更。把它当成”代码库概览”或”指向其他文件的索引”，而不是塞进所有细节。在 monorepo 中，给每个团队的目录单独配一份子目录 CLAUDE.md，并用 claudeMdExcludes 设置跳过与自己无关的团队文件。

示例 · 在项目根目录创建 CLAUDE.md

比如你有一个 C++/Python 混合的量化项目，记不住构建命令和目录结构，可以在项目根目录建一个 CLAUDE.md：

# 项目说明

## 构建命令
- C++ 部分：`cmake -B build && cmake --build build`
- Python 部分：`pip install -r requirements.txt --break-system-packages`
- 跑测试：`pytest tests/`

## 目录结构
- `src/cpp/`：HyperGraph 传播模块，C++ 实现
- `src/python/`：多模态抽取 pipeline
- `data/`：不要提交到 git，已加入 .gitignore

## 代码规范
- C++ 用 Google Style，缩进 2 空格
- Python 用 black 格式化
- commit message 用中文，格式：`类型: 简述`

存好之后，以后每次在这个项目里打开 Claude Code，它都会自动知道怎么编译、去哪找代码，你不用每次都重新解释一遍。

02 · Rules（规则）

.claude/rules/ 目录下的 markdown 文件，用来给 Claude 具体的约束或约定。

不限定路径的规则和 CLAUDE.md 行为一样：会话开始就常驻加载，压缩后会重新注入——即便这次任务根本用不上，也会白白占用上下文。

而路径限定的规则可以通过 paths 字段控制加载时机：例如限定在 src/api/** 的规则，在纯写文档的会话里根本不会出现在上下文中，只有当 Claude 读取该目录下的文件时才会加载。

---
paths:
  - "src/api/**"
  - "**/*.handler.ts"
---
所有 API handler 在处理前必须用 Zod 校验输入。
加载时机
全局：会话开始 / 路径限定：触碰匹配文件
压缩表现
压缩后重新注入
上下文成本
中等（不限路径则常驻）

建议：像”迁移文件只能追加、不能修改”这类针对特定文件的约束，最适合写成带 paths: frontmatter 的 rule。当某条指令涉及一个横跨多处（但不是全部）代码库的通用关注点或文件类型时，优先用路径限定规则，而不是在多个子目录里各放一份 CLAUDE.md。

示例 · 只在 C++ 目录生效的规则

你 C++ 记忆管理不熟，容易漏写析构或忘记 RAII，可以建一个 .claude/rules/cpp-memory.md：

---
paths:
  - "src/cpp/**"
  - "**/*.cpp"
  - "**/*.h"
---
写 C++ 代码时：
1. 优先用智能指针（unique_ptr / shared_ptr），不要手动 new/delete
2. 涉及资源管理的类，遵循 RAII，别忘了写析构函数
3. 每次改完提醒我跑一下 valgrind 或 ASan 检查内存泄漏

这样一来，你写 Python 的时候这条规则根本不会加载进上下文，只有碰到 .cpp/.h 文件才会生效——不用担心它跟 Python 部分的规则打架。

03 · Skills（技能）

.claude/skills/ 下的文件夹，包含指令、脚本和资源，Claude 会按需动态加载。每个 skill 有一个 SKILL.md，包含名称、描述和正文。

会话开始时只加载名称和描述，完整正文只有在 Claude 调用该 skill 时才加载——可以通过斜杠命令（如 /code-review）触发，也可以由 Claude 根据任务自动匹配。

Skills 是通过系统提示词中的名称+描述触发调用的。

例如内置的 /code-review 技能会审查当前 diff 并报告发现，不会直接改文件——这个 skill 定义了固定的检查流程，保证每次调用都遵循同一套结构化方法。

压缩发生时，Claude Code 会在一个”跨所有已调用 skill 的共享预算”内重新注入这些 skill；如果一次会话调用了很多 skill，最先被调用的（最旧的）会先被丢弃。

加载时机
名称+描述常驻 / 正文调用时加载
压缩表现
按共享预算重注入，最旧先丢
上下文成本
低（正文只在调用时加载）

建议：像部署流程、发布检查表、review 流程这类”操作性”指令，应该放进 skill，而不是塞进 CLAUDE.md。

示例 · 一个”跑量化回测”的 skill

你的量化平台每次回测都要走固定几步，手动敲容易漏步骤，可以建一个 .claude/skills/backtest/SKILL.md：

---
name: backtest
description: 运行多策略量化回测并生成收益/夏普/回撤报告
---
执行以下步骤：
1. 检查 data/events/ 下事件 JSON 是否为最新（对比日期）
2. 运行 `python backtest/run.py --config config/backtest.yaml`
3. 检查输出的年化收益、Sharpe、最大回撤是否达标：
   - 年化收益 15-25%
   - Sharpe 1.5-2.5
   - 最大回撤 ≤ 15%
4. 如果不达标，列出可能原因（数据缺失/参数问题/过拟合）
5. 生成简要报告，不要直接修改策略参数

写好之后，之后想跑回测直接在会话里说”帮我跑一下回测”，Claude 就会按这套固定流程走；平时不跑回测的会话里，这一整段完全不会占用上下文。

04 · Subagents（子代理）

.claude/agents/ 下的 markdown 文件，定义用于特定旁支任务的独立助手。每个文件用 YAML frontmatter（name、description，以及可选的 model、工具权限）加正文，正文会成为该 subagent 的系统提示词。

Subagent 和 skill 类似：会话开始只加载名称、描述、工具列表，正文不会自动触发。Claude 通过 Agent 工具调用它，传入一段 prompt。

Claude Code 的上下文窗口容纳了本次会话中 Claude 知道的一切。

关键区别在于：subagent 正文里那部分更大的指令内容，不仅不会自动触发，而且完全不会进入主会话的上下文。它在一个全新的、独立的上下文窗口中运行，只有它的最终消息（通常是多个子任务的汇总结果）加上一些元数据，会返回到主会话。

这种模式可以扩展：subagent 最多可以嵌套 5 层，动态工作流甚至可以在不需要你手动指定每个细节的情况下，编排数十到上百个后台 agent——编排计划和中间结果存在脚本变量里，而不是 Claude 的上下文窗口里，因此能做到扩展而不损失指令的准确性。

加载时机
名称/描述/工具列表常驻，正文调用时加载
压缩表现
只有最终摘要+元数据返回主会话
上下文成本
低（调用前零成本）

建议：这种”隔离性”正是选 subagent 而不是 skill 的核心原因。当一个旁支任务（深度搜索、日志分析、依赖审计）会产生一堆你之后不会再用到的中间结果、只会污染主会话时，用 subagent；如果你想让整个流程在主线程里展开、能实时看到并把控每一步，就用 skill。

示例 · 一个”文献检索”子代理

你之前查过 2025-2026 年多模态+量化金融的论文，这种搜索会产生一堆你不需要留在主会话里的中间结果（翻了哪些页面、排除了哪些不相关论文），适合建一个 subagent，存成 .claude/agents/paper-search.md：

---
name: paper-search
description: 检索多模态LLM与量化金融交叉领域的最新论文，只返回精选结果摘要
tools: WebSearch, WebFetch
---
你是一个论文检索助手。收到主题后：
1. 搜索近 12 个月内的相关论文（arXiv、KDD、AAAI 等）
2. 排除明显不相关的结果，不用在最终回复里解释排除过程
3. 只返回：论文标题、作者、一句话核心贡献、链接
4. 最多返回 5 篇，按相关度排序

在主会话里说”用 paper-search 帮我查一下视频理解+强化学习交易的最新论文”，Claude 会调用这个子代理，它自己在独立上下文里翻页、筛选，最后只把 5 条精选结果带回主会话——中间搜索过程占用的 token 不会污染你正在写代码的主线程。

05 · Hooks（钩子）

用户自定义的命令、HTTP 端点或 LLM 提示，通过在 Claude 生命周期的特定事件（如文件编辑、工具调用、会话开始）上触发，来实现对 Claude 行为更确定性的控制。

一次 Claude Code 会话中，各类事件在哪些节点可以触发 hook。

Hook 在 settings.json、受管策略设置，或 skill/agent 的 frontmatter 中注册。类型分为 command、HTTP、mcp_tool、prompt、agent 五种：前三种是确定性执行；后两种（prompt 和 agent）依靠 Claude 的判断而不是固定规则来决定输出。

Hooks 的上下文成本很低，因为配置或指令内容存在主上下文窗口之外——harness 直接运行 handler（command / http / mcp_tool），或者在独立的上下文窗口里发起模型调用（prompt / agent）。有些 hook 的输出会被存进主上下文（比如一个拦截型 hook 的报错信息，会保留在上下文里让 Claude 知道调用为什么被拒绝），但大多数 hook 除非配置里显式要求，否则不会把输出存进主窗口。

加载时机
生命周期事件触发时
压缩表现
完全绕过压缩机制
上下文成本
低（配置在主上下文外）

建议：任何”必须确定性发生”的事情都用 hook：编辑后自动跑 linter、任务完成后发 Slack 通知、执行前拦截特定命令。PreToolUse hook 可以检查任意工具调用，返回 exit code 2 即可拒绝执行。

示例 · 编辑 Python 文件后自动格式化

你在 macOS 上装了 homebrew，可以先装好格式化工具，再在 .claude/settings.json 里配一个 hook：

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "black \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

配好之后，Claude 每次改完一个文件，hook 会自动跑 black 格式化——不管 Claude 有没有”想起来”要格式化，这一步都会确定性发生，不依赖它记不记得你的规范。

06 · Output Styles（输出风格）

.claude/output-styles/ 下的文件，直接把指令注入系统提示词。它们永远不会被压缩掉，每次会话开始都会加载，并在会话内首次请求后被缓存——因此上下文成本属于中等。

因为它直接位于系统提示词中，output style 承载着目前所有方式里最高的指令遵循权重，应当谨慎使用。

修改 output style 会替换掉默认的 output style（除非在该 style 的 frontmatter 里设置 keep-coding-instructions: true）。在 Claude Code 中，这会移除告诉 Claude”你在协助软件工程任务”的默认指令，以及其他关键默认设定，比如：

  • 如何界定改动范围；
  • 什么情况下该加/该省代码注释；
  • 安全相关的处理方式；
  • 声明任务完成前的验证习惯（比如先跑测试）。

默认情况下，自定义 output style 会丢掉以上全部内容，Claude Code 会从”软件工程助手”变成更接近”通用助手”的角色。

加载时机
会话开始，注入系统提示词
压缩表现
永不压缩
上下文成本
高（但会覆盖默认系统提示词）

建议：写自定义 output style 之前，先看看内置的几种：Proactive（主动）、Explanatory（讲解）、Learning（学习协作）已经覆盖了大部分常见需求（自主性、教学模式、协作式编程），不需要你自己维护一份 style 文件。

示例 · 切换到讲解模式

你想让 Claude 在写代码的同时顺便帮你巩固操作系统/网络知识，不用自己写 style 文件，直接在会话里选内置的：

/config

在弹出的菜单里找到 Output style 这一项，选 Explanatory（讲解模式），回车确认。旧版本可以直接用 /output-style 命令切换，但这个命令已在新版本中被移除，现在统一走 /config 菜单。切换后不会立刻生效，要 /clear 或开新会话才会应用。选中之后，Claude 回答时会穿插”Insight”小段落，解释这段代码用了什么原理、为什么这么写。

07 · Append System Prompt（追加系统提示词）

修改 output style 之外的另一种方式，是 append-system-prompt 参数。相比修改 output style 文件可能带来大范围、非预期的行为变化，append 参数只是在原有系统提示词的基础上追加，不会改变 Claude 的角色定位，只是在默认角色上再加一些指令。

它是在调用时通过命令行参数传入的，只对那一次调用生效，不会像文件那样跨会话持久保存。

相比其他方式，append-system-prompt 的上下文成本可能更高：它会增加输入 token（不过会话内首次请求后可以靠 prompt caching 降低成本），如果你让 Claude 用更啰嗦或更长的风格回复，也会增加输出 token。

加载时机
会话开始，作为 CLI 参数传入
压缩表现
永不压缩，仅本次调用生效
上下文成本
中等(首次请求后被缓存)

建议：append-system-prompt 最适合用来加特定的编码规范、输出格式，或某个领域的专业知识。但要注意它对”遵循度”的边际效益是递减的：塞进去的指令越多，Claude 遵循得越不严格，尤其是当指令之间互相矛盾时。

示例 · 启动时临时追加一条要求

这个不是会话内命令，是启动 claude 时在 macOS 终端里加的参数。比如你今天想让所有回答都带时间复杂度分析：

claude --append-system-prompt "回答涉及算法时，一定要标注时间复杂度和空间复杂度"

只问一次就退出，不开交互会话，可以配合 -p：

claude -p "解释一下这段快排代码" --append-system-prompt "回答涉及算法时，一定要标注时间复杂度和空间复杂度"

内容长的话（比如一整套规范），先存成文件再引用：

claude --append-system-prompt-file ./my-rules.txt

它只对这一次启动生效，关掉终端下次又要重新加。如果发现自己天天都要加同一句话，说明该把它写进 CLAUDE.md 了，而不是每次手打参数。嫌麻烦的话，可以在 ~/.zshrc 里配个 alias：alias ccx='claude --append-system-prompt "回答简洁，命令行操作给完整命令"'，以后敲 ccx 就带上了。

四个常见误区

如果你发现自己正在做以下任何一件事，说明该换个地方放指令了。

✕ 误区
在 CLAUDE.md 里写"每次 X 都必须 Y"

如果这个行为需要可靠地发生（比如每次编辑后跑 prettier、完成后发 Slack 通知），应该用 settings.json 里的 hook，而不是指令。"模型选择去跑格式化工具"和"格式化工具自动运行"是两件完全不同的事。

→ 换成：Hook

✕ 误区
在 CLAUDE.md 里写"绝对不能做 X"

对于绝对不能发生的事情，纯指令是错误的工具。Claude 大部分时候会遵循指令，但在长会话、模糊场景，或者文件中被注入了 prompt injection 的情况下，模型可能不遵守被 prompt 出来的规则。真正的护栏需要是确定性的，能实现这一点的是 hooks 和权限系统。PreToolUse hook 可以检查调用并返回 exit code 2 来拦截；受管设置（managed settings）更进一步——由管理员部署、用户本地配置无法覆盖，是唯一能强制实现组织级确定性护栏的方式。

→ 换成：Hook + 权限系统（关键场景用受管设置）

✕ 误区
在 CLAUDE.md 里写一段 30 行的操作流程

流程性的东西应该放进 skill。CLAUDE.md 应该只放 Claude 需要一直记住的"事实"：构建命令、monorepo 布局、团队约定。部署手册或安全审查清单应该放进 .claude/skills/，正文只在被调用时才加载。

→ 换成：Skill

✕ 误区
写一条只适用于 API 目录的规则，却没加 paths 限定

如果一条规则只适用于 src/api/**，用 paths: 限定它可以让它在无关工作时不进入上下文。不加限定的规则，效果跟直接写进 CLAUDE.md 没区别：永远加载、永远消耗 token。

→ 换成：路径限定的 Rule

✕ 误区
把个人偏好写进项目级 CLAUDE.md

所有基于文件的方式都有一个"用户级"版本，会在你所有仓库的所有会话中加载，跟当前用的是哪个项目无关。个人偏好（比如"总是用语义化 commit message"）用本地/用户级文件；项目级文件留给"团队共用但和这个代码库强绑定"的约定。

小结：怎么选

一句话版本——事实（构建命令、目录结构）放 CLAUDE.md；约束（尤其是范围有限的）放 rules，能限路径就限路径；流程（部署清单、review 步骤）放 skills；需要隔离、只要结果的旁支任务用 subagent；**必须百分百执行、不容模型”选择”**的事情用 hook；大幅换角色才用 output style；临时的语气/格式调整用 append-system-prompt。等这些机制都跑起来之后，还可以把 skills、subagents、hooks、output styles 打包成一个 plugin，方便在团队或多个项目之间共享同一套配置。
